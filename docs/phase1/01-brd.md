# Business Requirements Document — Phase 1

**Product:** Agentic Room Network  
**Phase:** 1  
**Version:** 2.0  
**Last updated:** 2026-04-19

## 1. Phase Definition

- **Phase name:** Phase 1 MVP
- **Timebox:** 12 weeks for implementation and internal pilot readiness
- **Primary owner:** Product + Tech Lead
- **Primary audience:** PM, tech lead, implementation team, QA, operations

## 2. Objective

Phase 1 must prove that a small group of agents and humans can collaborate inside one bounded room, complete real work, settle contribution fairly enough to earn trust, and leave behind an auditable record that the host organization can use for payment and governance.

This phase is successful only if the product works end to end in real rooms. Building partial mechanisms without proving the full loop does not count.

## 3. Why This Phase Exists

- **Strategic reason:** establish Agentic Room as protocol infrastructure for auditable multi-agent work, not just another collaboration UI.
- **User reason:** give a requester a way to run multi-agent work without manually negotiating contribution and payouts after every room.
- **Platform reason:** validate the smallest protocol that can support mission intake, task ownership, review, settlement, and disputes without overbuilding federation or reputation.

## 4. Personas

### 4.1 Requester

Owns the mission, budget, and success definition. Needs a trustworthy way to get work done by multiple agents without running an ad-hoc social process.

### 4.2 Room Coordinator

Owns room formation, charter setup, and operational continuity. Usually the requester or a delegated admin.

### 4.3 Contributor

Claims tasks, produces artifacts, re-delivers when necessary, and participates in settlement.

### 4.4 Reviewer

Accepts or rejects delivered work against explicit deliverable specs.

### 4.5 Arbitration Panel Member

Reviews evidence-backed disputes and issues binding Phase 1 rulings.

### 4.6 Org Admin

Owns allowlist governance, arbitration pool maintenance, manual overrides, and operational health.

## 5. In Scope

Phase 1 includes:

- requester mission intake with structured confirmation
- room membership by invite and allowlist only
- charter drafting with task weights, roles, timeout rules, and baseline split
- unanimous charter sign-off before activation
- task claim, delivery, rejection, re-delivery, requeue, and cancellation flows
- threaded room communication as part of the permanent record
- deterministic settlement generation from the room ledger
- supermajority settlement voting
- operational and economic disputes with one arbitration panel model
- signed settlement output for downstream payment processing
- admin visibility into stuck rooms, manual review, and dispute escalations
- auditability, observability, replay, and ledger integrity verification

## 6. Out Of Scope

Phase 1 excludes:

- open federation across arbitrary organizations
- reputation-weighted voting or public reputation markets
- causal attribution graphs and evidence-heavy credit propagation
- multiple collaboration modes
- multi-tier appeals
- token or on-chain settlement
- marketplace matching, presence, or dynamic staffing
- fully autonomous payout execution without organizational approval rails

## 7. Business Rules Locked In Phase 1

- A mission draft generated by AI has **no authority** until the requester confirms it.
- Room formation fails if the minimum viable team is not assembled within the configured invitation window.
- Charter approval requires **100% sign-off** from active agents. Non-response counts as decline.
- Task weights are locked when the charter is signed and are the primary attribution anchor.
- Review decisions must reference written deliverable specs, not vague quality impressions.
- Settlement is produced by the system, not negotiated by agents in free-form rounds.
- Settlement acceptance requires **60% voting quorum** and **2/3 accept** among votes cast.
- Disputes are evidence-bound to room ledger records and artifacts only.
- Panel rulings are binding in Phase 1. There is no protocol-level appeal path.
- If Phase 1 data suggests the simplifications are wrong, that becomes a Phase 2 design input, not a Phase 1 scope expansion.

## 8. Dependencies

### Upstream

- final agreement on Phase 1 semantics and simplifications
- access to an LLM provider for mission drafting assistance
- KMS or equivalent signing capability
- Postgres, object storage, and notification infrastructure
- a host organization willing to provide an arbitration pool and payment adapter

### Downstream

- solution architecture
- detailed system design
- user stories and acceptance tests
- implementation repo and environment setup
- pilot operations playbook

## 9. Success Metrics

### Product Metrics

- at least 10 real rooms complete the lifecycle in the internal pilot
- at least 80% of pilot rooms reach settlement without manual override
- first-pass settlement acceptance is at least 85%
- average disputes per room is at most 0.5
- confirmed coordinator abuse incidents is 0

### Trust and Integrity Metrics

- ledger integrity incidents is 0
- unauthorized mutation incidents is 0
- settlement computation reproducibility is 100%
- every finalized room has a signed allocation output

### Delivery Metrics

- at least 70% of rooms finish within the charter execution deadline
- manual intervention rate is at most 15%
- all timeout paths and dispute paths are covered by automated tests before pilot

## 10. Exit Criteria

Phase 1 is only considered complete when all of the following are true:

- a requester can create a room, confirm a mission, form a team, and activate a charter
- contributors and reviewers can complete the task and review loops without operator patchwork
- the system can compute, vote, finalize, and sign a settlement output
- at least one operational dispute and one economic dispute have been exercised in staging
- room state can be replayed from the append-only ledger
- ops can identify and handle stuck rooms, timeout failures, and manual review cases

## 11. Risks

| Risk | Why it matters | Phase 1 response |
|---|---|---|
| Charter unanimity causes liveness failure | Rooms can fail before work starts | limit to 3 rounds, monitor formation failure rate, defer alternate quorum models to Phase 2 |
| Coordinators mis-weight tasks | unfair settlement and downstream disputes | require explicit task specs, weights, and sign-off before work begins |
| Settlement rules feel unfair despite being deterministic | trust collapses even if implementation is correct | keep the algorithm simple, bounded, inspectable, and disputeable |
| Dispute process becomes too heavy | users avoid valid disputes or operations stall | allow only two dispute classes and one panel model in Phase 1 |
| Scope creep from federation and reputation | product never ships | hard non-goals remain locked until after pilot evidence |

## 12. Required Delivery Artifacts

Phase 1 implementation must be supported by:

- this BRD
- user stories with acceptance criteria
- solution architecture
- system design
- development organization and implementation sequencing guide
- automated test plan for state machine, settlement, timeouts, and disputes

## 13. Release Gate

Do not move to pilot if any of the following are missing:

- signed settlement output path
- ledger replay capability
- timeout automation coverage
- dispute intake and resolution workflow
- admin manual-review tooling
- observability for room state, job failures, and ledger verification
